# VCS Skip, systemd Unit, Rule Expansion & Housekeeping Bundle Design

Date: 2026-07-02
Status: Approved

## Goal

Clear four independent improvements in one pass: stop watching version
control internals (`.git` and friends), ship a systemd user unit for
supervised operation, add verified rules for four more ecosystems
(Composer, Elixir, Zig, Dart), and round out `Cargo.toml` metadata and the
release profile.

## Background

The 2026-07-02 follow-up survey found the codebase healthy but identified
these gaps:

- Every directory under `.git/` (e.g. `objects/xx`) gets an inotify watch.
  A Dropbox tree with many git repositories wastes thousands of watches
  against `max_user_watches` and suffers event storms during git
  operations, even though nothing inside `.git` ever matches a rule.
- Watch mode is designed to fail fast so "a supervisor can restart it"
  (README), yet the repository ships no supervisor configuration.
- Composer, Elixir, Zig, and Dart artifact directories are unhandled.
  Directory semantics were verified against current official docs via
  Context7: Composer's `vendor` is regenerated by `composer install` and
  recommended against committing; Mix's `_build`/`deps` are regenerated by
  `mix compile`/`mix deps.get`; Zig's `zig-out`/`.zig-cache` are build
  output and compiler cache; Dart's `.dart_tool` is documented as "never
  check into source control, safe to delete".
- `Cargo.toml` lacks `description`/`readme`, and no `[profile.release]`
  tuning exists for distributed binaries.

## Non-Goals

- Marking `.git` with the Dropbox ignore attribute (syncing repositories
  stays the user's choice; we only stop watching their internals).
- A template (`@`) systemd unit or a `--scan-once` timer pair (single
  fixed-path user unit was chosen for KISS).
- Legacy Zig `zig-cache` (undotted, renamed to `.zig-cache` in Zig 0.13).
- A shutdown session-summary log (descoped: threading a marked-path
  counter through the event loop is not worth the value; `journalctl`
  already shows per-path info logs).
- unmark, rule configuration, and multi-root watching (remain in the
  backlog).

## Design

### 1. VCS directory watch skip

- New `MatchAction::SKIP_ONLY` constant:
  `{ set_dropbox_ignore: false, skip_descendants: true }`.
- `ArtifactDirsRule` gains an `action: MatchAction` field. All existing
  constants keep `IGNORE_AND_SKIP`; a new `VCS_DIRS` constant
  (`.git`, `.hg`, `.svn`, `.jj`, `.bzr`) uses `SKIP_ONLY`. A separate rule
  type was rejected because `matches` would be duplicated verbatim.
- No changes to `walk()` or `plan_entry()`: on a skip-only match the walk
  neither records a match nor pushes the directory as a watcher (the
  `skip_descendants` `continue` runs before the watcher push), and the
  event path sets `watch_dir = false`, so a newly created `.git` is never
  watched either.
- `RuleMatch::log_matched` logs at `debug` when `set_dropbox_ignore` is
  false and `info` otherwise, so skip-only matches do not flood the info
  log (one line per repository directory otherwise).
- Behavior notes: paths created inside `.git` are no longer observed at
  all (nothing in there ever matched a rule); a `.git` *file* (submodule /
  worktree) fails the `is_dir` check and is unaffected; `--scan-once`
  shares the walk and gets faster for free.

### 2. systemd user unit

- New `contrib/dropignore.service` (approved content):

  ```ini
  [Unit]
  Description=Mark build artifacts as Dropbox-ignored
  After=default.target

  [Service]
  ExecStart=%h/.local/bin/dropignore %h/Dropbox
  Restart=on-failure
  RestartSec=5

  [Install]
  WantedBy=default.target
  ```

- README gains a "Running as a service" section: copy the unit to
  `~/.config/systemd/user/`, adjust the binary/Dropbox paths,
  `systemctl --user daemon-reload && systemctl --user enable --now
  dropignore`, inspect logs with `journalctl --user -u dropignore`, and a
  pointer to `max_user_watches` for large trees.

### 3. Rule expansion, round 3

| Rule | Mechanism | Marker (= rescan trigger) |
|---|---|---|
| Composer `vendor` | `MarkedBuildDirRule::COMPOSER_VENDOR` | `composer.json` |
| Elixir `_build` | `MarkedBuildDirRule::MIX_BUILD` | `mix.exs` |
| Elixir `deps` | `MarkedBuildDirRule::MIX_DEPS` | `mix.exs` |
| Zig `zig-out` | `MarkedBuildDirRule::ZIG_OUT` | `build.zig` |
| Zig `.zig-cache` | `ArtifactDirsRule` (exact name) | — |
| Dart `.dart_tool` | `ArtifactDirsRule` (exact name) | — |

- Generic names (`vendor`, `deps`, `_build`, `zig-out`) are marker-guarded
  so unrelated trees never match (e.g. Go's committed `vendor/` has no
  `composer.json`). Markers automatically become rescan triggers via
  `Rule::triggers`.
- Dotted tool-owned names (`.zig-cache`, `.dart_tool`) follow the
  `.gradle`/`.terraform` precedent: exact-name match, no marker. They join
  new `ArtifactDirsRule` constants (e.g. `ZIG_CACHES`, `DART_CACHES`) with
  `IGNORE_AND_SKIP`.
- All new constants are registered in `RuleEngine::new` in `src/app.rs`;
  the README feature list is updated.

### 4. Cargo metadata & release profile

- `Cargo.toml` `[package]`: add `description` and `readme = "README.md"`.
- New `[profile.release]`: `strip = true`, `lto = "thin"` to shrink
  distributed binaries.

## Implementation order

1 (VCS skip) → 3 (rules) → 4 (metadata) → 2 (systemd unit + README). Items
are independent; the two rules.rs items run first so the README edits in 2
land after the feature list is final. `./scripts/check.sh` gates every
step.

## Testing

- Item 1: unit tests for `VCS_DIRS` (matches `.git` dir, not a `.git`
  file); walk test asserting `.git` appears in neither `watchers` nor
  `matches` while its siblings still do; event-path test asserting a newly
  created `.git` directory is not watched and a match created inside it is
  not marked.
- Item 3: per-rule match/non-match unit tests (marker present vs absent)
  and a trigger-registration assertion for `composer.json`, `mix.exs`,
  `build.zig`.
- Item 4: `cargo build --release` succeeds.
- Item 2: no code; README/unit reviewed by inspection.
- Whole bundle: `cargo test`, `cargo clippy --all-targets` (zero
  warnings), `cargo fmt --check` via `./scripts/check.sh`.
